# 커피 주문 앱

## 1. 프로젝트 개요

### 1.1 프로젝트 명
커피 주문 앱

### 1.2 프로젝트 목적
사용자가 커피 메뉴를 주문하고, 관리자가 주문을 관리할 수 있는 간단한 풀스택 웹 앱

### 1.3 개발 범위
- 주문하기 화면(메뉴 선택 및 장바구니 기능)
- 관리자 화면(재고 관리 및 주문 상태 관리)
- 데이터를 생성/조회/수정/삭제할 수 있는 기능

## 2. 기술 스택
- 프런트엔드: HTML, CSS, 리액트, 자바스크립트
- 백엔드: Node.js, Express
- 데이터베이스: PostgreSQL

## 3. 기본 사항
- 프런트엔드와 백엔드를 따로 개발
- 기본적인 웹 기술만 사용
- 학습 목적이므로 사용자 인증이나 결제 기능은 제외
- 메뉴는 커피 메뉴만 있음

## 4. 주문하기 화면 PRD (UI·UX)

### 4.1 화면 목적
- 사용자가 커피 메뉴를 탐색하고, 옵션을 선택한 뒤 장바구니에 담아 주문을 확정할 수 있게 한다.
- 상단 내비게이션으로 **주문하기**와 **관리자** 화면을 구분하되, 본 문서 범위는 **주문하기** 탭이 활성화된 상태의 화면만 다룬다.

### 4.2 레이아웃 구조
화면은 세 영역으로 구성한다.

| 영역 | 위치 | 역할 |
|------|------|------|
| 헤더 | 상단 전체 | 브랜딩 및 화면 전환 |
| 메뉴(상품) 목록 | 헤더 아래 본문 | 카드 형태로 메뉴 노출 및 담기 |
| 장바구니 | 하단 | 담은 상품·금액 요약 및 주문 확정 |

### 4.3 헤더
- **로고/타이틀:** 좌측에 앱명 **COZY** 표시.
- **탭(또는 링크 버튼):** 우측에 **주문하기**, **관리자** 두 항목.
  - 현재 화면이 주문하기일 때 **주문하기**에 시각적 강조(예: 테두리·배경 등 활성 상태).
  - **관리자** 선택 시 관리자 화면으로 이동(라우팅/전환). UI 상세는 **§5**를 따른다.

### 4.4 메뉴 카드(상품 단위)
각 메뉴는 카드 하나로 표현하며, 가로로 나란히 배치한다(반응형 시 줄 바꿈 허용).

- **이미지:** 상단에 정사각형 비율의 썸네일 영역. 이미지가 없을 때는 플레이스홀더(와이어프레임상 X 박스 등)로 대체 가능.
- **상품명:** 굵은 글씨(예: 아메리카노(ICE), 아메리카노(HOT), 카페라떼).
- **가격:** 기본 가격을 원화 형식으로 표시(예: 4,000원). 천 단위 구분 기호 사용.
- **설명:** 짧은 한 줄 설명 영역(예: “간단한 설명…”). 실제 카피는 데이터로부터 채운다.
- **옵션:** 체크박스 목록. 각 항목은 라벨에 옵션명과 **추가 금액**을 표시한다.
  - 예: 샷 추가 (+500원), 시럽 추가 (+0원)
  - 체크 시 해당 메뉴의 **이번에 담을 한 줄**에만 옵션이 반영되도록 한다(카드 단위 로컬 선택 상태).
- **담기 버튼:** 라벨 **담기**. 클릭 시 현재 카드에서 선택된 옵션 조합으로 장바구니에 한 줄 추가한다.

### 4.5 장바구니 영역
- **제목:** 좌상단 **장바구니**.
- **줄 항목:** 각 줄은 다음을 포함한다.
  - 상품명과, 적용된 옵션을 괄호 등으로 표기(예: 아메리카노(ICE) (샷 추가)).
  - 수량(예: X 1, X 2).
  - 해당 줄의 **소계**(옵션 포함 단가 × 수량).
- 옵션이 없는 경우에는 상품명과 수량만 표시해도 된다(예: 아메리카노(HOT) X 2).
- **총 금액:** 장바구니 내 모든 줄의 소계 합. 라벨 **총 금액**과 함께 원화로 표시. 항목이 바뀔 때마다 실시간으로 갱신한다.
- **주문하기 버튼:** 최종 확정 액션. 라벨 **주문하기**(헤더 탭과 동일 문구이나 역할이 다름: 여기서는 **주문 제출**).
  - 클릭 시 주문 생성 API(또는 약속된 백엔드 계약) 호출. 성공/실패 피드백은 구현 단계에서 토스트·모달 등으로 정하면 된다.

### 4.6 주요 사용자 시나리오
1. 사용자가 메뉴 카드에서 옵션을 체크하고 **담기**를 누른다 → 장바구니에 동일 조합이 없으면 새 줄 추가, 동일 상품·동일 옵션 조합이면 수량 증가로 처리할지 여부는 구현 시 정책으로 확정(와이어프레임은 별도 줄로 표시된 예시에 맞춤).
2. 여러 메뉴를 담는다 → 장바구니 목록과 총 금액이 누적 반영된다.
3. **주문하기**(하단)를 눌러 주문을 제출한다.

### 4.7 프런트엔드 상태·데이터 요구사항(요약)
- 메뉴 목록: 상품 ID, 이름, 기본 가격, 설명, 이미지 URL(선택), 옵션 정의(이름, 추가 금액).
- 장바구니 줄: 상품 참조, 선택된 옵션 집합, 수량, 줄별 소계(파생값).
- 총 금액: 줄 소계의 합(파생값).
- 헤더의 활성 탭: 현재 라우트/뷰와 동기화.

### 4.8 범위 밖·제약(기존 기본 사항과의 정합)
- 결제·로그인은 다루지 않는다.
- 재고 차감·주문 상태 변경은 관리자·백엔드 영역에서 처리하며, 주문하기 화면 PRD는 **주문 요청 전송 및 UI 일관성**까지를 범위로 한다.

## 5. 관리자 화면 PRD (UI·UX)

### 5.1 화면 목적
- 관리자가 **주문 흐름 요약**, **메뉴별 재고**, **들어온 주문 목록**을 한 화면에서 확인하고, 재고를 조정하며 주문 상태를 진행시킬 수 있게 한다.
- 상단 내비게이션은 주문하기 화면과 동일하되, 본 문서 범위는 **관리자** 탭이 활성화된 상태다.

### 5.2 레이아웃 구조
헤더 아래 본문을 세 블록으로 쌓는다(위에서 아래 순).

| 영역 | 역할 |
|------|------|
| 관리자 대시보드 | 주문 건수 요약(상태별 집계) |
| 재고 현황 | 메뉴 단위 재고 수량 및 증감 조작 |
| 주문 현황 | 개별 주문 상세와 상태 전환 액션 |

주문 건수가 많을 경우 **주문 현황**은 세로 스크롤 가능한 목록으로 구현한다.

### 5.3 헤더
- **로고/타이틀:** 좌측 **COZY**(주문하기 화면과 동일).
- **탭:** 우측 **주문하기**, **관리자**.
  - 관리자 화면일 때 **관리자**에 시각적 강조(와이어프레임상 테두리 박스 등).
  - **주문하기** 선택 시 주문하기 화면으로 이동(**§4**).

### 5.4 관리자 대시보드
- **섹션 제목(권장):** 관리자 대시보드 또는 동등한 구역 라벨.
- **요약 한 줄:** 아래 지표를 한 줄(또는 한 블록)에 나란히 표시한다. 값은 서버 집계 또는 클라이언트가 목록을 기준으로 계산해 **화면 진입 시·주문/상태 변경 후** 최신으로 맞춘다.

| 지표 | 의미 |
|------|------|
| 총 주문 | 시스템에 존재하는 주문 전체 건수(또는 조회 범위 내 전체 — 구현 시 API 계약에 맞춤) |
| 주문 접수 | 아직 제조로 넘기지 않은 대기 상태 주문 건수 |
| 제조 중 | 제조 진행 중인 주문 건수 |
| 제조 완료 | 완료 처리된 주문 건수 |

와이어프레임 예시 값: 총 주문 1, 주문 접수 1, 제조 중 0, 제조 완료 0.

### 5.5 재고 현황
- **섹션 제목:** 재고 현황.
- **구성:** 메뉴(상품)마다 동일한 형태의 카드 또는 행을 반복한다.
- **표시 항목**
  - **상품명**(예: 아메리카노 (ICE), 아메리카노 (HOT), 카페라떼).
  - **현재 수량:** 숫자 + 단위 **개**(예: 10개).
  - **조작:** **[+]** 재고 1 증가, **[-]** 재고 1 감소.
- **동작:** 버튼 클릭 시 백엔드에 재고 갱신을 요청하고, 성공 시 화면 수치를 갱신한다. 0 미만으로 내려가지 않게 할지 여부는 정책으로 정하되, 학습용 앱에서는 **0에서 [-] 비활성화 또는 무시**를 권장한다.
- 표시 대상 메뉴는 주문하기 화면의 메뉴 집합과 일치하는 것이 자연스럽다.

### 5.6 주문 현황
- **섹션 제목:** 주문 현황.
- **주문 한 건(행/카드)에 포함할 정보**
  - **주문 시각:** 사용자에게 익숙한 형식(예: 7월 31일 13:00). 데이터는 UTC 저장·로컬 표시 등 구현 선택.
  - **주문 내용 요약:** 품목·수량(예: 아메리카노(ICE) x 1). 여러 품목이면 줄바꿈 또는 쉼표로 나열.
  - **금액:** 해당 주문 합계(예: 4,000원). 천 단위 구분.
  - **액션 버튼:** 와이어프레임상 **주문 접수** — 의미는 **이 주문을 다음 단계(제조 중)로 전환**한다. 현재 상태가 “주문 접수”일 때만 이 라벨을 쓰거나, 상태에 따라 **제조 시작**, **제조 완료** 등으로 바꾸는 **상태 기반 버튼**으로 확장해도 된다(와이어는 1단계만 명시).

### 5.7 주문 상태 모델(프런트·백엔드 정합)
- 최소 상태 예시(이름은 API와 통일):
  1. **주문 접수** — 신규 주문, 대기.
  2. **제조 중** — 접수 처리 후.
  3. **제조 완료** — 픽업/완료 처리 후.
- 대시보드의 네 지표는 위 상태에 매핑된 건수 집계와 일치해야 한다.
- 주문하기에서 제출된 주문은 기본적으로 **주문 접수**로 생성되는 것을 전제로 한다(**§4**와 연동).

### 5.8 주요 사용자 시나리오
1. 관리자가 화면을 연다 → 대시보드 숫자, 재고 카드, 주문 목록이 로드된다.
2. 재고 **[+]** / **[-]**로 수량을 조정한다 → 서버 반영 후 UI가 갱신된다.
3. 신규 주문 행에서 **주문 접수**(또는 다음 단계 버튼)를 누른다 → 해당 주문 상태가 바뀌고, 대시보드 집계·목록 표시가 갱신된다.

### 5.9 프런트엔드 상태·데이터 요구사항(요약)
- 대시보드: 상태별 주문 건수(또는 전체 주문 목록으로부터 파생 계산).
- 재고: 상품 ID, 이름, 현재 재고 수량; 증감 API.
- 주문 목록: 주문 ID, 생성 시각, 품목 요약, 총액, 현재 상태; 상태 변경 API.
- 헤더 활성 탭: 현재 라우트와 동기화.

### 5.10 범위 밖·제약
- 관리자 **로그인·권한 구분**은 기본 사항(§3)에 따라 범위 밖. 화면은 URL로 진입 가능한 수준을 전제로 해도 된다.
- 결제·배달 추적은 다루지 않는다.
- 재고가 0일 때 주문하기 화면에서 주문을 막을지는 **주문하기·백엔드 정책**으로 별도 합의(본 PRD는 관리자 측 재고 표시·편집에 초점).
